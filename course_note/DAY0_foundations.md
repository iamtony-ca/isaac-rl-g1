## 0. 딱 하나의 비유로 시작 — "게임 속에서 걷기 배우기"

강화학습(RL)은 **강아지 훈련**이나 **자전거 배우기**와 같다: 정답을 알려주는 게 아니라, **해보고 → 잘하면 칭찬(+) 못하면 벌(−) → 다음엔 더 잘하게** 조금씩 고치는 것.

문제는 실제 로봇으로 이걸 하면 수만 번 넘어져 부서진다. 그래서 **시뮬레이션(게임 세계)** 안에서 연습한다.
시뮬은 두 가지가 강력하다:
1. **안전하고 공짜** — 넘어져도 리셋하면 그만.
2. **빠르고 병렬** — GPU로 **로봇 수천 개를 동시에** 연습시켜, 실물의 수천 배 속도로 경험을 모은다.

> 이 노트 전체의 그림: **게임 속에 내 로봇 4096마리를 복제해 동시에 걷기 연습시키고, "코치"가 점수를 매겨 로봇의 "뇌"를 조금씩 고친다.**

### 1-4. "왜 config가 이렇게 많지?" — Manager-based의 정체
Isaac Lab은 연습장을 **레고 블록 조립**처럼 만든다:
- "관찰은 이 항목들로", "보상은 이 항들의 합으로", "행동은 관절 목표각으로" …를 각각 **Manager(담당자)** 에게 맡긴다.
- 그 담당자들에게 넘기는 **설정표**가 `~Cfg`(config). **로직이 아니라 파라미터 시트**다.
- 그래서 DAY4에서 본 것처럼 "reward weight"만 바꾸면 코드 없이 행동이 바뀐다.

| 용어 | 비유 | 한 문장 직관 |
|---|---|---|
| **Manager-based** | 레고 블록 조립식 환경 | 관찰/행동/보상/종료를 모듈로 끼워 조립 |
| **Cfg / config** | 설정표(주문서) | "이 로봇, 이 보상, 이 명령범위로" 적은 파라미터 |
| **gym / task id** | 연습장 이름표 | `Isaac-Velocity-Flat-G1-v0` = 특정 config를 부르는 이름 |
| **Direct workflow** | 조립식 대신 직접 코딩 | 유연하지만 손이 많이 감(우리 강좌는 Manager-based) |

## 2. RL의 기본 루프 — "감각 → 결정 → 행동 → 점수"

매 순간(50Hz = 초당 50번) 이게 돈다:
```
   관찰(무엇을 느낀다) → 정책(어떻게 할지 결정) → 행동(관절을 움직인다) → 보상(점수를 받는다)
        ↑                                                                      │
        └──────────────────────  다음 순간, 반복  ←──────────────────────────┘
```
- **정책(policy)** = 로봇의 **뇌**. "이 관찰이면 이 행동" 하는 결정기. 실체는 **신경망**(숫자 in → 숫자 out).
- **학습** = 받은 점수를 보고 **뇌(신경망)를 조금씩 고쳐** 다음엔 더 높은 점수를 받게 하는 것.

| 용어 | 비유 | 한 문장 직관 |
|---|---|---|
| **Agent (에이전트)** | 학습하는 주체 | 정책+학습 알고리즘 = "배우는 로봇" |
| **Policy (정책)** | 로봇의 뇌/직관 | 관찰 → 행동. 신경망 하나 |
| **Neural network** | 숫자를 변환하는 필터 뭉치 | 입력(123) → 은닉층 → 출력(37) |

## 3. rsl_rl의 PPO 알고리즘 — 개념적으로 깊게

PPO는 "뇌를 **어떻게** 고칠까"의 방법. 다섯 조각으로 나눠 이해하자.

### 3-1. 정책은 "주사위를 던진다" (탐험)
- 뇌는 딱 하나의 행동이 아니라 **행동의 확률분포**(평균 μ + 퍼짐 σ)를 낸다.
- **왜?** 처음엔 뭐가 좋은지 모르니 **조금씩 다르게 시도(탐험)** 해봐야 뭐가 좋은지 알 수 있다.
- `init_noise_std=1.0` = 처음 퍼짐. 학습되며 σ가 줄어 → "이제 확신 있으니 덜 흔든다".
- 배포(추론) 땐 σ를 버리고 **평균 μ만** 씀 → 결정론적(DAY6).

> 비유: 초보 운전자는 핸들을 이리저리 시험(큰 σ)하다, 익숙해지면 정확히 조금씩만 튼다(작은 σ).

### 3-2. "코치"가 상황을 평가한다 (Critic / Value)
- **Critic(가치망 V)** = 옆에서 "지금 이 상황, 앞으로 얼마나 좋아?"를 점치는 **코치**.
- 실제 신경망 하나 더(actor와 별개). 학습에만 쓰고 배포엔 안 씀.
- **왜 필요?** 행동이 좋았는지 판단하려면 "기대치"가 있어야 하니까.

### 3-3. Advantage — "기대보다 좋았나?" (학습의 핵심 신호)
- **Advantage = 실제로 받은 점수 − 코치가 예상한 점수.**
  - `+`(기대보다 좋음) → **그 행동의 확률을 높여라**
  - `−`(기대보다 나쁨) → 그 행동을 줄여라
- 이게 뇌를 어느 방향으로 고칠지 정하는 **나침반**.
- **GAE(γ, λ)** = 이 Advantage를 똑똑하게 계산하는 법. 당장의 점수와 먼 미래 점수를 **얼마나 섞을지**의 균형.
  - `gamma=0.99` = 미래를 얼마나 중시(≈2초 앞까지), `lam=0.95` = 추정의 안정성 조절.

> 비유: 시험을 봤는데 코치가 "넌 70점 예상"이라 했다. 85점 나오면(+15, advantage +) "그 공부법 좋네, 더 해"; 60점이면(−) "그 방법 줄여".

### 3-4. On-policy — "지금 이 뇌로 겪은 경험만 쓴다"
- PPO는 **현재 뇌**가 직접 겪은 경험으로만 배운다(=on-policy). 남의(옛날 뇌) 경험은 못 씀.
- 그래서 흐름이: **① 지금 뇌로 잠깐 굴려 경험 수집(rollout) → ② 그걸로 뇌 조금 수정 → ③ 경험 버림 → 반복.**
- **그래서 로봇 수천 마리가 필요하다** — 매 라운드 신선한 경험을 충분히 모으려고.

| 용어 | 비유 | 직관 |
|---|---|---|
| **rollout / 수집** | 연습 경기 한 세트 | 현재 뇌로 N스텝 굴려 데이터 모음 |
| **on-policy** | "내가 직접 뛴 것만" | 현재 정책의 경험만 학습에 사용 |
| **iteration** | 한 라운드(수집+수정) | rsl_rl 로그의 "iteration N" |

### 3-5. PPO clip — "한 번에 너무 확 바꾸지 마" (PPO의 심장)
- 어떤 행동이 좋아 보이면 그 확률을 왕창 올리고 싶다. 하지만 **적은 경험만 보고 과하게 바꾸면 학습이 망가진다.**
- PPO는 한 번 수정에서 **행동 확률이 옛 뇌 대비 ±약 20%(`clip_param=0.2`)** 넘게 안 바뀌게 **잘라낸다(clip)**.
- 이 "조금씩만 바꾼다"는 **안전벨트**가 PPO가 안정적인 이유.

> 비유: 한 번 시험 잘 봤다고 공부법을 통째로 갈아엎지 말고, **조금만 조정**해라. 표본이 적으니 과신 금지.

### 3-6. 곁가지 두 개
- **Entropy(엔트로피) 보너스** `0.008` = "약간의 무작위성/호기심을 유지" → 너무 일찍 한 행동에 굳지 않게(DAY4의 "얼어붙은 로봇" 반대 방향).
- **KL + adaptive lr** = rsl_rl이 매 수정 후 **뇌가 얼마나 바뀌었나(KL divergence)** 를 재서, 너무 많이 바뀌면 **학습 속도(lr)를 자동으로 낮춘다**. → 사람이 lr을 손으로 안 맞춰도 된다(rsl_rl이 편한 이유).

## 4. 한 iteration을 통째로 따라가기 (실제 숫자로)

우리가 DAY3에서 돌린 학습 한 라운드:
```
1) 수집:  512마리 로봇 × 24스텝 = 12,288개 (관찰,행동,보상,advantage) 경험
2) 학습:  그 경험을 4조각(minibatch)으로 나눠, 5번 반복(epoch)하며 뇌 수정
             → 매번 Advantage로 방향 정하고, PPO clip으로 조금씩만, KL 보며 lr 조절
3) 폐기:  경험 버림(on-policy) → 새 뇌로 다시 1)
```
- **epoch(5)** = 모은 경험을 몇 번 우려먹나(너무 많이 우리면 그 경험에 과적합).
- **minibatch(4)** = 한 번에 다 말고 작은 덩어리로 나눠 학습(안정·효율).
- 이 라운드(iteration)를 flat은 1500번, rough는 3000번 반복 = 본 학습.

## 부록 — 더 깊이 (심화 개념 7)

### A. Manager-based — "폼에 체크해서 조립하는 환경"
Isaac Lab의 환경은 하나의 큰 프로그램이 아니라, **역할별 담당자(Manager)** 들의 모임이다:
| Manager | 담당 | 넘기는 Cfg |
|---|---|---|
| ObservationManager | 로봇이 뭘 느낄지 | `ObservationsCfg`(각 항: base_lin_vel, joint_pos …) |
| ActionManager | 행동을 관절에 어떻게 적용할지 | `ActionsCfg`(JointPositionAction) |
| RewardManager | 점수 항들의 합 | `RewardsCfg`(track_lin_vel, feet_air_time … 각 weight) |
| TerminationManager | 언제 판을 끝낼지 | `TerminationsCfg`(넘어짐, 시간초과) |
| CommandManager | 명령(목표 속도) 생성 | `CommandsCfg`(속도 범위) |
| EventManager / CurriculumManager | 랜덤화 / 난이도 상승 | `EventCfg` / `CurriculumCfg` |

- 각 Manager는 **"term(항)"의 리스트**를 들고 있다. 예: RewardManager에 `feet_air_time` term = 함수 + 파라미터 + weight.
- **핵심 이점**: 행동을 **코드가 아니라 config(폼 체크)로** 바꾼다 → DAY4에서 weight만 만진 이유, 로봇 갈아끼우기 쉬운 이유(CUSTOM_ROBOT).
- 반대는 **Direct**: `step()`을 직접 코딩. 유연하지만 손이 많이 감.
- *비유: Manager-based = 커스텀 피자 주문서(소스·토핑 칸 체크). Direct = 반죽부터 직접.*

### B. Critic / Value — "코치는 왜 필요한가"
- **Value V(s)** = "이 상태에서 앞으로 받을 총점의 기대치". 이걸 점치는 **두 번째 신경망**(actor와 별개).
- **왜 필요?** advantage(아래)를 구하려면 "이 정도면 보통"이라는 **기준선(baseline)** 이 있어야 한다.
- 기준선이 없으면 그냥 "실제로 받은 총점"으로 학습하는데, 이건 한 판의 운에 따라 **엄청 출렁(고분산)** 해서 학습이 불안정.
- critic이 학습된 기준선을 주면 **분산이 확 줄어** 안정적으로 배운다. critic은 실제 받은 총점으로 회귀 학습되고, **배포 땐 버린다**(actor만 씀, DAY6).
- *비유: 시험 점수를 "잘 봤나?"로 판단하려면 코치의 "예상 점수"가 있어야 한다. 없으면 매번 절대점수에 일희일비.*

### C. GAE(γ, λ) — advantage를 "똑똑하게" 추정
**Advantage A(s,a)** = "이 상태에서 이 행동이 평균보다 얼마나 좋았나". 학습의 나침반(+면 그 행동↑). 문제는 **어떻게 추정하냐**:
- **몬테카를로**(끝까지 실제 총점 − V): 정확(무편향)하지만 한 판 운에 출렁(**고분산**).
- **1-step TD**(r + γV(s′) − V(s)): 안정(**저분산**)하지만 V가 틀리면 같이 틀림(**편향**).
- **GAE(λ)** = 이 둘을 **λ로 섞는** 지수가중 평균: λ=0이면 1-step(저분산·편향), λ=1이면 몬테카를로(무편향·고분산), **λ=0.95 = 긴 시야를 보되 약간 부드럽게**(대부분 여기).

두 손잡이를 헷갈리지 말 것:
- **γ(gamma, 0.99) = "미래를 얼마나 멀리 보나"(시야).** 100스텝 뒤 보상은 0.99¹⁰⁰≈0.37배로 할인 → **유효 시야 ≈ 100스텝(2초)**. γ↑ = 예지력↑ 이지만 학습 어렵고 노이즈↑.
- **λ(lambda, 0.95) = advantage "추정"의 편향-분산 절충**(시야가 아니라 계산 방식의 부드러움).
- *비유: γ = 체스에서 몇 수 앞을 내다보나. λ = 내 직감(코치 예측)을 얼마나 믿고 vs 실제 결과가 나올 때까지 기다리나.*

### D. On-policy — "지금 이 뇌로 겪은 것만"
- PPO는 **현재 정책**이 만든 데이터로만 배운다. 업데이트가 "지금 정책 대비 얼마나 바꿀까"(ratio·advantage)라서, 정책을 한 번 고치면 **옛 데이터는 낡아**(다른 정책이 만든 것) 못 쓴다.
- 반대는 **off-policy**(DQN/SAC): 옛 경험을 저장통(replay buffer)에 쌓아 재사용 → 샘플 효율↑.
- **대가와 이유**: on-policy는 매 iteration **신선한 데이터를 새로 모아야** 해서 비효율적. 하지만 **안정적**이고, 시뮬에선 **로봇 4096마리 병렬**로 데이터를 싸게 왕창 모을 수 있어 이 단점이 상쇄된다. → **PPO+대규모 병렬 시뮬**이 짝인 이유.

### E. rollout — "한 판 수집분"
- **rollout** = 한 iteration에서 현재 정책으로 굴려 모은 **경험 묶음**(각 스텝의 관찰·행동·보상·value·확률).
- rsl_rl: `num_steps_per_env(24) × num_envs` 개. = "collection(수집) 단계"의 산출물.
- 이걸로 학습(아래)한 뒤 **버린다**(on-policy). 다음 iteration은 새 rollout.

### F. epoch / minibatch — "모은 걸 어떻게 우려먹나"
한 rollout으로 그래디언트를 **한 번만** 밟지 않는다:
- rollout을 섞어 **`num_mini_batches`(4)개 조각**으로 나눠 조각마다 한 번씩 업데이트 = **minibatch**(안정·GPU 효율).
- 이 전체 통과를 **`num_learning_epochs`(5)번 반복** = **epoch**. → iteration당 4×5 = **20 그래디언트 스텝**.
- **왜 5번만?** 여러 번 우리면 정책이 rollout을 만든 정책에서 점점 멀어져(=데이터가 낡아짐) **KL이 튀고** clip이 싸우기 시작 → 과하면 해롭다. 그래서 **3~5가 보통**. (여기서 clip/KL이 왜 필요한지와 연결됨)
- *비유: 문제집 한 권(rollout)을 4단원(minibatch)으로 나눠 5회독(epoch). 근데 너무 여러 번 회독하면 그 문제집에만 최적화(과적합).*

### G. flat / rough — "같은 태스크, 다른 난이도"
| | Flat | Rough |
|---|---|---|
| 지형 | 평지(plane) | 계단·경사·박스·요철(생성) |
| 센서 | proprioceptive만 | + **height-scan**(발밑 187점) |
